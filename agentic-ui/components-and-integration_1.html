Section 01 — Component Catalog

The building blocks each framework gives you.

A2UI ships an abstract component catalog (the "Basic" set in v0.9) that agents reference by type. CopilotKit ships React/Angular hooks and pre-built UI components that wrap your own design-system widgets. The table below pairs comparable primitives.

Capability
A2UI · Basic Catalog
CopilotKit · React API

Text & Headings

Display text, titles, and rich-text content.

Component types Text with usageHint: "h1"
Inline Markdown for rich text.
  • Bound via literalString or { "path": "/data/x" }
  • Streamed incrementally via JSONL
React pattern Native JSX — <h1>, <Markdown />, or any component you ship.
  • Streamed via AG-UI messageDelta events
  • Use <CopilotChat /> for chat-mode rendering

Form Inputs

Text fields, date pickers, selects, checkboxes.

Component types TextField, DateTimeInput, Dropdown, Checkbox.
  • Bound to a Data Model via JSON-Pointer paths
  • Two-way binding: input updates local model
  • Agent only reads on submit (privacy-by-default)
React pattern Your own form components, registered via useCopilotAction.
  • Parameters declared with type + description for the LLM
  • Schema enforced at the runtime boundary

Buttons & Actions

User-triggered events sent back to the agent.

Component types Button with an action: { name } payload.
  • Action sent over A2A or AG-UI back to agent
  • Optional data captured from bound paths
React pattern useCopilotAction({ name, parameters, handler }).
  • Direct function-call from agent to your handler
  • Optional render for inline UI
  • Optional renderAndWaitForResponse for HITL

Layout & Containers

Cards, columns, lists, tabs.

Component types Card, Row, Column, List, Tabs.
  • Flat adjacency-list structure (children referenced by ID)
  • LLM-friendly — easy to stream incrementally
React pattern Your design system, unchanged. CopilotKit doesn't impose layout.
  • Pre-built <CopilotSidebar />, <CopilotPopup />
  • Headless option for full layout control

Media

Images, video, file embeds.

Component types Image, Video components.
  • Client-side URL sanitization recommended
  • Use trusted-domain proxies to prevent SSRF
React pattern Multimodal inputs first-class: file uploads, voice, transcripts.
  • Persistent storage in Threads (Enterprise tier)
  • Voice with auto-transcription

Custom Components

Your own widgets — charts, maps, dashboards.

BYOC pattern Define a Catalog with Zod schemas. Register via the host's renderer; description is injected into the agent prompt.
  • One catalog, many platforms (Web, Flutter, Angular)
React pattern Pass any React component into render in useCopilotAction.
  • Agent sends args; component renders with them
  • Real-time updates as agent streams state

State Sharing

Agent ↔ application synchronized state.

Data Model A2UI defines a separate Data Model object. Components bind via JSON-Pointer paths (e.g. /reservation/date).
  • Server-pushed updateDataModel messages
  • Strict upsert semantics
useCoAgent / useAgent Bidirectional state hook — your React state is the agent's state.
  • Read-only or read-write modes
  • Streamed event-sourced diffs with conflict resolution

Human-in-the-Loop

Approvals, edits, mid-run pauses.

Pattern Agent emits an approval surface (e.g. a Card with two Buttons). User action returns to agent over the transport.
  • HITL is possible but not a first-class primitive
renderAndWaitForResponse Built-in HITL primitive. Agent pauses; UI awaits a respond() call.
  • Approve / reject / edit flows out of the box
  • Captured in persistent Threads
Read this carefully

A2UI lists component types. CopilotKit lists developer hooks.

This asymmetry reflects the different layers each operates at. A2UI is the wire format your agent speaks. CopilotKit is the framework your React app uses to consume that wire format (and others). In the recommended pattern from the executive brief, you write CopilotKit hooks on the client while your agents emit A2UI on the server — the two catalogs above are complementary tools, not duplicate ones.

client app · BookingPanel.tsx
TSX 
import { useCopilotAction } from "@copilotkit/react-core";
import { FlightCard } from "./components";

// Verbatim shape from the official CopilotKit landing page.
export function BookingPanel() {

  // 1. Register an action the agent can call.
  //    The agent learns about it from `name` + `description`.
  useCopilotAction({
    name: "showFlight",
    description: "Display a flight card for the user",
    parameters: [
      { name: "flight",  type: "object", required: true },
      { name: "status",  type: "string" }
    ],

    // 2. The agent's call renders YOUR React component.
    render: ({ args }) => (
      <FlightCard
        flight={args.flight}
        status={args.status}
      />
    )
  });

  // 3. (optional) Human-in-the-Loop: pause for user approval.
  useCopilotAction({
    name: "confirm_booking",
    parameters: [
      { name: "booking", type: "object", required: true }
    ],
    renderAndWaitForResponse: ({ args, status, respond }) => (
      <ConfirmationCard
        booking={args.booking}
        isExecuting={status === "executing"}
        onCancel={() => respond?.({ approved: false })}
        onConfirm={() => respond?.({ approved: true })}
      />
    )
  });

  return <CopilotSidebar />;
}

# From a2ui.org/guides/agent-development
from google.adk.agents.llm_agent import Agent
from a2ui_sdk import A2uiSchemaManager, BasicCatalog

# 1. Initialize the schema manager with the Basic Catalog.
#    This builds a JSON Schema the LLM is constrained to.
schema_manager = A2uiSchemaManager(
    version=VERSION_0_9,
    catalogs=[BasicCatalog()]
)

# 2. Compose the system prompt with rules + schema + examples.
AGENT_INSTRUCTION = """
You are a restaurant finding assistant.
Your final output MUST be an A2UI UI JSON response,
following the schema and rules below.
"""

# 3. Define the agent. The LLM is free to choose any
#    component from the trusted catalog — and ONLY those.
root_agent = Agent(
    model="gemini-2.5-flash",
    name="restaurant_agent",
    description="Finds restaurants and books tables.",
    instruction=AGENT_INSTRUCTION + schema_manager.prompt(),
    tools=[get_restaurants],
)

# 4. Validate every LLM output against A2UI_SCHEMA
#    before sending it on the wire (defence in depth).
jsonschema.validate(
    instance=parsed_json,
    schema=schema_manager.a2ui_schema_object
)

Section 03 — Integration Flow

From user prompt to rendered widget.

Step-by-step lifecycle of a single user request, traced through each framework's runtime. Both end at the same place — pixels on the screen — but they take meaningfully different paths to get there.

A2UI · Lifecycle

01
User prompts the agent

Plain text: "Book me a table for 2 tomorrow at 7."

02
Agent generates A2UI JSON

LLM produces a JSONL stream conforming to the A2UI_SCHEMA. Schema validation is performed.

03
Transport delivers the message

A2A protocol or AG-UI carries each JSONL line via SSE.

04
Client renderer parses the payload

The renderer walks the flat component list, builds the tree from id references, and resolves data bindings.

05
Native widgets are mounted

Component types from the trusted catalog map to React / Flutter / Lit / Angular implementations. No code execution.

06
User interaction → action message

Click sends a userAction A2A message back. Agent may reply with surfaceUpdate or updateDataModel.

CopilotKit · Lifecycle

01
User prompts the copilot

Via <CopilotChat /> or <CopilotSidebar /> — or programmatically via useAgent().

02
Runtime forwards to the backend agent

CopilotRuntime routes the message to LangGraph, Mastra, ADK, or whichever agent is configured.

03
Agent emits AG-UI events

Stream of typed events: messageDelta, toolCall, stateUpdate, uiSurface (which can carry A2UI).

04
React hooks dispatch the events

useCopilotAction matches by name; useCoAgent updates shared state in place.

05
Your React components render

The hook's render function returns JSX with the typed args. A2UI surfaces are handled by the A2UI renderer.

06
State persists in the Thread

Enterprise tier: full interaction trace (UI, HITL, files, voice) saved to the Thread for cross-session resume.

Section 04 — Developer Touchpoints

Where the engineer's hands actually go.

The contact surface is small in both frameworks once you see it laid out. Here are the primitives an engineer touches in a typical day's work.

A2UI

Server-side primitives · Apache 2.0
  • A1 A2uiSchemaManager Server-side helper that builds the LLM system prompt with the schema, examples, and your custom catalog.
  • A2 BasicCatalog The default component set. You ship a custom catalog for your domain components (charts, maps, etc.).
  • A3 JSONL message envelopes createSurface, surfaceUpdate, dataModelUpdate, deleteSurface.
  • A4 JSON-Pointer data bindings Components reference state via paths like /reservation/date; client manages the model.
  • A5 Renderer per platform Drop in the official renderer for React, Angular, Lit, or Flutter. Wraps your own widgets.
  • A6 Schema validation jsonschema.validate() on every agent output catches malformed UI before it reaches the wire.

CopilotKit

Client-side React/Angular hooks · MIT
  • C1 <CopilotKit> provider Wrap your app once with runtimeUrl. All hooks underneath get the connection.
  • C2 useCopilotAction() Register an action with name, parameters, and a render function — the agent gets a tool, the user gets UI.
  • C3 useCoAgent() / useAgent() Bidirectional state sync between React and the agent. Modern useAgent is the v2 superset.
  • C4 useCopilotReadable() Expose any piece of app state to the agent as readable context. No prompt-engineering required.
  • C5 renderAndWaitForResponse First-class HITL: the agent pauses, your component captures the user's decision, then respond() resumes it.
  • C6 Pre-built UI: <CopilotSidebar> Drop-in chat UI with deep CSS + sub-component customization. Or go fully headless with useCopilotKit().
Section 05 — Recommended Architecture

The recommended combined stack.

As recommended in the executive brief: adopt CopilotKit with A2UI rendering enabled. The four-stage architecture below is what that looks like in production — each layer with its concrete technology choice.

01 · BACKEND
Agent Reasoning

The agent that thinks, plans, and uses tools to gather information or perform actions on the user's behalf.

LangGraph · ADK · Mastra · MAF
02 · PAYLOAD
UI Description

Agent emits a declarative component-tree that describes intent. Schema-validated, no executable code.

A2UI v0.9 · JSONL
03 · TRANSPORT
Bidirectional Stream

Carries A2UI surfaces, tool calls, state diffs, HITL pauses, and voice/file events between agent and frontend.

AG-UI Protocol · SSE / WebSocket
04 · CLIENT
Rendering & State

React app consumes the stream, renders A2UI surfaces with your native components, and manages shared state.

CopilotKit React SDK
Why this combination is the right answer

Each layer plays to its strength.

A2UI gives you cross-platform portability and security-by-design at the payload layer. AG-UI gives you a production-tested transport with guardrails. CopilotKit gives you the React SDK, persistence, and enterprise governance. Used together, no layer is overburdened, and no layer is reinvented in-house. This is the architecture Oracle, Microsoft, and Google have publicly aligned around.
