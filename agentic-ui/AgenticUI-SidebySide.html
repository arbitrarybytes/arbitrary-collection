The four-layer stack — and where these two toolkits fit

CopilotKit and A2UI are not competitors — they sit at different layers and are complementary in production. This showcase demonstrates each in isolation so the trade-offs are visible.

How to read this showcase

  • Each scenario tab shows the same agent task implemented twice.
  • Left pane = CopilotKit (typed React tool component).
  • Right pane = A2UI (declarative JSON tree against a catalog).
  • The shared event log below each scenario shows the AG-UI events both panes consume — same protocol, different rendering.
  • At the bottom of each scenario, the code comparison shows the actual integration patterns.

The key insight for executives

  • AG-UI is the wire. Both toolkits ride on the same event protocol — adopting one does not lock you out of the other.
  • CopilotKit wins today for React-only enterprise web apps with strong typing and fast iteration.
  • A2UI wins tomorrow when the platform must reach mobile/desktop or be rendered by partner-owned clients.
  • Recommended pattern: CopilotKit chrome + A2UI payloads inside generative-UI events as A2UI matures past v0.9.

const { state, setState } = useCoAgent<ConfigState>({
  name: "configure-agent",
  initialState: { chassis: "X14", ram: 16, price: 1299 }

// User edits → setState → patch streams to agent.

  onChange={e => setState({ ...state, ram: +e.target.value })}>

// Agent recomputes price → state arrives back automatically.

// Agent emits state patch only:

Adoption Verdict — for an enterprise e-commerce platform

Distilled from the five scenarios above and the AgenticUI reference document.

CopilotKit · ship today

Recommended for: phase 1–3 of platform rollout (next 1–2 quarters).
  • GA, Apache-2.0, broad runtime support (LangGraph, MAF, ADK, AWS, OpenAI)
  • First-class generative UI, CoAgents, HITL, agent steering
  • Type safety end-to-end (Zod / JSON-Schema → tool → component)
  • Highest-fidelity React UX; hand-tuned per domain
  • Best fit for Shop, Account, Support today
  • React-only on the open path (Angular WIP)
  • Each tool needs its own component; cross-platform requires re-shipping per renderer
  • Less natural for dense reactive forms (Configure)

A2UI · adopt strategically

Recommended for: Configure canvas surfaces and any non-web client (mobile/desktop/partner).
  • Framework-agnostic (React, Flutter, SwiftUI, Compose) from one agent payload
  • Catalog-as-security: components are pre-approved; agents cannot define new ones
  • Data-path bindings make dense reactive forms cheaper than per-permutation components
  • Native fit for the Configure canvas and partner-embedded surfaces
  • v0.9 spec — re-validate before standardizing platform-wide
  • Less expressive than hand-written React for one-off rich UX
  • Requires you to maintain a robust component catalog up front
Recommended hybrid stack
Wire protocol
AG-UI (open, Apache-2.0, broad runtime support)
Frontend toolkit
CopilotKit for the chrome (chat, sidebar, HITL, CoAgents)
Rendering — Shop/Account/Support
Tool-as-component via useCopilotAction
Rendering — Configure/Canvas/Partner
Declarative widget JSON; migrate to A2UI when v1.0 lands
Runtime
LangGraph (TS or Python) or Microsoft Agent Framework
Patterns supported
Sequential · Concurrent · Group chat · Handoff · Magentic
Security boundary
Zod / Pydantic structured outputs at tool layer; component catalog whitelist at UI layer; AG-UI interrupts on every write action
Bottom line: these are not rival vendors — they are layers of the same stack. Adopt CopilotKit + AG-UI now; add A2UI as a payload format inside generative-UI events when the platform extends to mobile/desktop or partner clients.
